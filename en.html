Introduction

In this project, I wanted to answer the question: is it possible to write a game in Java for Sega Mega Drive / Genesis. I don't want to reveal any spoilers, but the answer is yes.

A few years ago, I came across the project, which allows you to write code for various retro processors in Java, including the Sega Mega Drive. Basically, it interprets the bytecode from the .class files obtained after compilation into 68K Assembly code. If the class file needs other class files, they are also read and processed. All API method calls are written in the output code, either as inline assembly code or as calls to pre-written functions that perform their task.

The system itself is quite simple, but I still have a lot to learn, and it is not so easy to search for high-quality information. In fact, this project was my first time doing real programming for Mega Drive.

  1.  Graphics

To learn how to display something on the screen, you need to understand the structure of graphics on the Sega platform and how to encode it. You can learn more about this in this article. In short, Sega uses tile graphics, where each tile is 8x8 pixels in size and takes up 32 bytes in ROM memory. A VDP(Video Display Processor) chip is used to display the image on the screen.

Data in VDP is loaded in a specific format that is closely related to the palette. The essence of this encoding is to assign each tile pixel a specific color index from the palette, which can vary from 0 to 15(0x0-0xF). Let's demonstrate this  approach using the example of the character Lemming from the game Lemmings Return for Mobile. This character is the smallest one I know, which uses all 16 colors of the palette and fits completely on just two tiles. 3.1 PALETTE

Let's continue our discussion of graphics and look at how the palette is stored in Java Grinder. This is important for understanding how other graphic elements work.

The Sega Mega Drive uses a 9-bit palette. You can read more about this here or here. Inconsole memory, one color palette takes up 2 bytes. For example, the white color value will be stored as 0x0E, 0xEE.

In Java Grinder, the palette is stored in the and loaded using the SegaGenesis VDP CRAM("Color RAM").

The palette array contains the color values of the 9-bit palette in 16-bit format from 0x000 to0xEEE. The maximum number of elements in the array must not exceed 16. If you use fewer colors, we recommend setting unused colors to 0x000.

Example of the lemming palette from previous section:

public static short[] palette =

  {  

    0xECE, 0x0A0, 0x0C0, 0x080, 0xEEE, 0x88C, 0xAAE, 0x246,

    0x8AE, 0x68C, 0x66A, 0xE80, 0xEA0, 0xC60, 0xC40, 0xA00 

  };

The palette value can be converted from RGB to 9 bit using this algorithm

((color.B >> 5) << 9) | ((color.G >> 5) << 5) | ((color.R >> 5) << 1).

Unfortunately, the exact inverse transformation is almost impossible to get, since there are 32 RGB colors per value. You can use this code for reverse conversion, but it does not guarantee that you will get the same colors as on the emulator or hardware.

b = (color9bit >> 9) & 0x7;

g = (color9bit >> 5) & 0x7;

r = (color9bit >> 1) & 0x7;

Color = (r << 5, g << 5, b << 5);

If you want to accurately convert a 9-bit palette to RGB, you need to find a matching table or output it yourself.

3.3 Sprites

Sprites are created very similar to the background, but they require more API calls to render. In addition, sprites do not contain a tile map, so they are not optimized, unlike the background. This means that the same sprite tiles can occur several times in VDP. In fact, this can be optimized, but this topic is beyond the scope of this article, you can read about it here.

Sprites are drawn in a 512x512 pixel virtual space, where the coordinates (128,128) coincide with the upper-left corner of the TV screen.

Inside the console, sprites are rendered in reverse order, i.e. from top to bottom, from left to right.

Example:

To display a sprite on the screen, we need to use the API functions:

  1. SegaGenesis.setPaletteColorsAtIndex(int index, short[] palette)

the function similarly.setPaletteColors(short[] palette) function, which is used to load the background palette, the only difference is that you can set the index of the start of loading the palette. The index value must be between 0 and 63. If you pass the index outside the range, it may lead to unforeseen consequences.

  1. SegaGenesis.setPatternTableAtIndex(int index, int[] patterns);

The function similarly.setPatternTable(int[] pattern) function. The index parameter defines the address from which tiles are loaded into video memory(VDP). It is not recommended to write to the range [0x0460, 0x0479], because the font is loaded in these addresses.

  1. SegaGenesis.setSpritePosition(int index, int x, int y);

The function adjusts the sprite position by the sprite index fromSprite Attribute Table, not to be confused with the index from the setPatternTableAtIndex function. For the sprite to appear on the screen, the x and y values must be in the range x=(128, 448) y=(128, 352).

  • SegaGenesis.setSpriteConfig1(int index, int value);

this is the so-called first word of the sprite configuration, which includes: the horizontal size of the sprite in tiles, the vertical size of the sprite in tiles, and the index of the next sprite to display.

  • SegaGenesis.setSpriteConfig2(int index, int value);

The second word that includes: palette number, horizontal or vertical display(optional), sprite address in VDP.

5. Sounds

In order to play at least some melody on the Sega Mega Drive, you need to know how the sound works on the platform. short, for audio playback on Sega is used: z80 CPU, z80 RAM, Yamaha 2612, PSG, Audio Mixer. We can only interact directly with the z80 RAM, and it will directly control everything else. An approximate interaction scheme looks like this:

Let's start by preparing the music and sound files. The music file should be monaural, with a sound depth of 8 bits and, preferably, with a sampling rate of 44100 Hz. recommend using files with the extension .wav, since this format contains all the necessary information in the header, and the data is stored in the original (RAW) format.[a]

API methods are used to work with z80:

  1. loadZ80(byte[] code): Loads code up to 8 kilobytes in size into Z80 RAM. Z80 will be reset via the API and the loaded code will start executing.
  2. resetZ80 (): Resets the Z80 state, returning it to its initial state.
  3. pauseZ80(): Pauses execution of Z80. This is necessary so that the m68000 host processor can access any resources in the Z80 space.
  4. startZ80 (): Starts Z80 execution again after it has been paused or stopped.

There are 3 ways to play a melody: load the sound in z80 RAM plus the initialization code, load the notes and their sequence in z80 RAM plus the initialization code, load the data from the wav file in rom and wav player in z80 RAM and pass the start address to a certain offset in z80 RAM. Let's analyze each method in order

Method 1. Load audio with the initialization code in z80 RAM.

First, we need the initialization code. Initialization code is z80 assembler code that is passed through the M68000 processor to z80 RAM as a compiled data array. To get it, you can compile the z80_play_dac file using naked_asm, translate it into a java byte array and extract the initialization code from it, or you can use a ready-made array of initialization code:

After inserting the initialization code into the z80 RAM, you have 512-58=454 bytes of free space, which is usually enough for a small sound effect, but not for playing the full melody.

Method 2: Write down notes or melodies and play them according to the specified scenario

The author of Java Grinder used guitar chords to implement this method and played them in a loop in the specified order. You can modify this code to create your own melody, then compile it using naked_asm and translate it into a java byte array.

Method 3: Write your own player or use a ready-made one.

For this method, you need to: place the player code in z80 RAM, place music data in ROM, determine the address and length of music data in ROM,write the address and length to a specific location in z80 RAM.

Unfortunately, here we face one of the limitations of Java Maximum size of a static array should not exceed 8,242 elements. We can overcome this limitation by using the int type instead of the byte type (the largest data type in Grinder at the moment), then we get that the maximum file size that can be loaded into a single array is 8242 * 4 = 32968 bytes, or almost 33 Kilobytes. This is exactly the length of a music file that we can load into ROM without any problems. To load it into ROM memory, we just need to refer to it, for example: byte[] b = z80_code or create an empty function in the file where you have the z80_code array and just call it, for example: public static void init(){}. If this size is not enough for you, you will have to create several arrays and call them all in turn in the wrapper function. Unfortunately, recording music to ROM one-to-one will not work, as 4 bytes will be inserted between the arrays, indicating the size of the array.

At the moment, there is no example of a custom wav file player, and there is also no melody that can be played using it.

8. Limitations

Here are the engine limitations that I encountered during development.

  1. You can't create objects if the new keyword isn't available
  2. You can't leave an empty if condition.
  3. You can't assign an initial value to enum. You can create enums, but you can't use them.
  4. Class names must be static final or without final, but then without initialization.
  5. you can't use addresses in VDP in the range [0x0460, 0x0479], because the font is located there;
  6. The commandsSegaGenesis. setPalettePointer(17);SegaGenesis.setPaletteColor(0x000); are not clear why they are needed. maybe they don't work
  7. The maximum size of a static array must not exceed 8 242 elements.
  8. Only large English letters are available in the font, WITHOUT NUMBERS OR PUNCTUATION MARKS.
  9. Only three types of numbers are supported: byte, short, int.
  10. You can't initialize a field in a method
  11. You can't access a char [] array element. For example, you can't write chr_arr[0]
  12. You can't play music and sound effects at the same time.
